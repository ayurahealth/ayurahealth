## 2025-05-04 - Timing Attack on Payment Signature & Insecure PRNG
**Vulnerability:** The Razorpay payment integration in `app/api/razorpay/create-order/route.ts` used a standard string equality operator (`===`) to verify the payment signature (`expectedSignature === razorpay_signature`). Additionally, the `receipt` ID generation used `Math.random()`, which is not cryptographically secure, to generate order receipts.
**Learning:** Standard string equality (`===`) evaluates character-by-character and returns `false` as soon as a mismatch is found. This leaks timing information, which could potentially be used by an attacker to forge a valid signature via a timing attack, especially for financial transactions where signature verification is critical. Using `Math.random()` to generate financial identifiers like receipt IDs introduces predictability, which could potentially be exploited to predict or brute-force order IDs.
**Prevention:** Always use `crypto.timingSafeEqual` with `Buffer.from` to compare sensitive hashes, signatures, and tokens in constant time, ensuring `length` checks are performed first to prevent TypeErrors. For generating critical identifiers, especially in financial contexts, prefer a cryptographically secure random number generator (CSPRNG) like `crypto.randomUUID()`.
