## 2024-11-28 - Secure Financial Receipt Generation and Signature Verification
**Vulnerability:** The application was using `Math.random()` to generate Razorpay order receipts, which is not cryptographically secure and could potentially lead to receipt collisions or predictability. Additionally, the Razorpay signature verification used a simple string equality check (`===`), making it vulnerable to timing attacks where an attacker could deduce the correct signature character by character by analyzing the response time.
**Learning:** For any critical identifier, especially those tied to financial transactions (like order receipts), cryptographically secure pseudorandom number generators (CSPRNG) must be used. `crypto.randomUUID()` is a good choice in a Node.js context. When comparing security tokens or signatures, standard equality operators short-circuit, causing the comparison time to depend on the number of matching characters. `crypto.timingSafeEqual` with `Buffer.from` should be used to guarantee constant-time execution, but it's critical to ensure both buffers are the same length before comparison to avoid runtime errors.
**Prevention:** Always use `crypto.randomUUID()` or `crypto.randomBytes()` for generating random tokens or IDs. Whenever comparing hashes, secrets, passwords, or signatures, strictly employ `crypto.timingSafeEqual`. Enforce length checks and handle null/undefined inputs explicitly before calling buffer functions to prevent server crashes.
